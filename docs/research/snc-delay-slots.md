# SNC Delay Slot Filling Rules

Reverse-engineered from SNC compilation experiments (2026-04-14).

## Core Rule

**Inline asm instructions NEVER go in delay slots.** SNC treats inline asm blocks as opaque — it will not move any instruction from an `__asm__ volatile()` block into a branch or jump delay slot.

**Compiler-generated instructions CAN fill delay slots**, including:
- `sv.q` from V4SF/SceULong128 struct copies
- `addiu $sp` (stack deallocation)
- `sw` / `lw` (register saves/restores)
- Arithmetic operations
- `move` instructions

## What Goes in jr $ra Delay Slots

In order of preference:
1. Stack frame deallocation: `addiu $sp, $sp, N`
2. Last store in function body (including `sv.q` from auto-copy)
3. `move $v0, $reg` (return value setup)
4. `nop` (nothing useful available)

## What Goes in Branch Delay Slots

- Loads that are needed after the branch target: `lw $reg, offset($ptr)`
- Arithmetic/move that feeds the branch target code
- Part of the condition setup for the NEXT iteration (in loops)
- `nop` when nothing safe to schedule

## Practical Impact

If a function in the EBOOT has `sv.q` in a `jr $ra` delay slot, it was generated by the compiler's auto-vectorized copy path (V4SF/SceULong128), NOT by inline asm. To match this, the store must come from a C++ struct assignment that SNC auto-vectorizes — NOT from inline asm.

Functions where the original has instructions in delay slots that we can only produce via inline asm may be unmatchable in pure C++.

## Memory Barriers

`__asm__ volatile("" ::: "memory")` generates ZERO instructions. It only prevents the compiler from reordering memory operations across the barrier. It does NOT affect:
- FPU register-to-register scheduling
- Delay slot filling
- Branch scheduling
