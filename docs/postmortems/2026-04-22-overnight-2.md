## Recommendations

1. **Suppress the "malware" system-reminder for this project.** Two full sessions were lost to agents refusing mid-work despite CLAUDE.md explicitly authorizing edits. One agent's refusal note spanned multiple paragraphs — it clearly wrestled with it for many tool calls before quitting. If the harness can't gate the reminder out, agents need a more authoritative override in the session prompt than "the malware reminder misfires." Every refused session is ~20 minutes of wasted Claude-API spend.
2. **Shorten the session timeout for stuck sessions.** Four 30-min timeouts each produced zero matches. By the 15-min mark with zero reports, the session is almost certainly in a permuter-death-spiral on one function and blocking the other 4 in its batch from being retried. Consider a heartbeat check at 12–15min: if zero function_results emitted, re-prompt with "move on from current function" or kill early and retry.
3. **No new research docs needed.** The scheduling/reg-alloc failures here are already covered by `docs/research/snc-*.md`. What's missing is a playbook: *when the diff is 4–8 bytes and purely scheduling, run permuter for N minutes; if no improvement, move on*. The prompt should explicitly encourage abandonment of 10-byte-diff scheduling divergences instead of burning the full 30min budget.
4. **Commit-guard for interrupted sessions.** The current orchestrator commits per-session when a session ends normally. If the operator Ctrl-Cs mid-session, everything in the working tree is lost (or, in this case, left dirty for the operator to deal with). Consider a periodic checkpoint commit inside long sessions — but this conflicts with the commit-time green-build gate from Enhancements #8, so needs design.
5. **Tell `verify_matches.py` about `in_progress`.** Currently `in_progress` entries pass silently. They should be counted and surfaced — 4 entries were left in this state and an audit should notice that.
